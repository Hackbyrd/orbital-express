Versioning: MAJOR.MINOR.PATCH (Ex. v2.3.2)

package.json Versioning on update
~1.2.3 will match all 1.2.x versions but will miss 1.3.0.
^1.2.3 will match any 1.x.x release including 1.3.0, but will hold off on 2.0.0.

Important: ALWAYS install the EXACT version of every node module. NEVER use tilde (~) or
caret (^) ranges. This applies to BOTH regular dependencies AND devDependencies — no exceptions.

  >> yarn add <MODULE_NAME> --exact          (regular dependency)
  >> yarn add <MODULE_NAME> --exact --dev     (dev dependency)

Why: range versions (~1.2.3, ^1.2.3) let npm/yarn silently pull a different version on a fresh
install, which means "works on my machine" bugs and non-reproducible builds. An exact pin (1.2.3)
guarantees every developer, CI run, and deploy installs the identical dependency tree.
Open package.json — every entry is an exact version with no ~ or ^. Keep it that way.

--------------------------------- Conventions ---------------------------------
-------------------------------------------------------------------------------

1. All code variables are camelCase
2. Database Table names are plural and PascalCase
3. Database Table Column names are camelCase
4. Database ENUM datatype names are ALL CAPS and NO SPACES OR UNDERSCORES OR DASHES
5. URLs for api requests have no spaces, no dashes, no underscores and no camelCase — even for
   multi-word actions: "/v1/users/logoutall" (NOT logout_all / logout-all / logoutAll), "updateemail",
   "smssendcode". Mirror the action name lowercased (V1LogoutAll -> logoutall). Why: predictable,
   consistent URLs and no hyphen-vs-underscore bikeshedding; a minor side effect is the run-together
   words are slightly less legible to casual crawlers. (Security still comes from auth, not URL spelling.)
6. URLs for pages will be dashes and lowercase
7. Folder/File names can have dashes but must be lowercase. The only exception is feature folders (folders under the "app" folder)
8. Feature folder names are singular and PascalCase (e.g. "Order", not "orders" or "Orders")
8a. Always scaffold via the generator (yarn gen / yarn del) — never hand-create or rm feature files.
    yarn del automatically removes the entry from actions/index.js / tasks/index.js; rm does not,
    leaving a broken export pointing at a deleted file. Immediately after scaffolding, remove the generator's default placeholder files
    using yarn del: `yarn del <Feature> -a V1Example`, `yarn del <Feature> -t V1ExampleTask`, and
    remove tests/helper.test.js. Never use rm for any of these.
9. Controllers are plurals
10. Actions are singular
11. All helpers/constants.js variables are UPPER CASE with underscores and NO DASHES
11b. No magic strings: any string literal used (or likely to be reused) in more than one place
     -- statuses, types, roles, locales, enum-like values -- lives in helpers/constants.js and is
     referenced (e.g. LOCALE.EN, not 'en'). Use the add-constant skill. EXCEPTION: migrations stay
     literal (frozen historical snapshots; add a "// = SOME.CONSTANT" comment instead of importing).
12. Boolean columns MUST start with a linking verb so they read as yes/no: is, has, can, does, should
    (isActive, hasFailed, canInviteOthers, doesRequireApproval). NEVER a bare noun (connectionsOnly -> isConnectionsOnly)
13. Foreign key columns are named <entity>Id (camelCase) and reference the real PascalCase plural table's id
    (userId -> Users(id)). Multiple FKs to the same table get a role prefix (hostUserId, bookerUserId -> Users(id))
14. Third-party vendor IDs prepend the vendor name (stripeId, twilioId, auth0Id)
15. Column order in a table: (1) primary key id, (2) foreign keys, (3) third-party vendor IDs,
    (4) customized columns, (5) Sequelize autogenerated: deletedAt, createdAt, updatedAt. See database/schema.sql template.

-------------------------------------------------------------------------------
----------------------- Action & Task Naming Convention -----------------------
-------------------------------------------------------------------------------

Formula: V{version}{ActionName}[By{Role}][On{Device}]

Segments:
  V{version}   - Always first. API version number. (V1, V2, V3...)
  {ActionName} - PascalCase. What the action does. (Create, Update, Login, Query...)
  By{Role}     - Optional. Add when behavior differs by user role.
  On{Device}   - Optional. Add when behavior differs by client platform.

Actions vs Tasks:
  Actions = real-time API requests that return a response immediately.
  Tasks   = background jobs that run in a worker process.
  RULE: Tasks always append "Task" after the action name.

  V1Create           -> action (real-time)
  V1CreateTask       -> task   (background job)

Role suffix (By{Role}):
  Only add when behavior genuinely differs per role.
  Shared logic between roles goes in helper.js — do NOT couple methods together.

  V1Create              no role distinction
  V1CreateByAdmin       admin only
  V1CreateByUser        user only
  V1CreateByAdminManager  admin with a specific role

Device suffix (On{Device}):
  Only add when behavior genuinely differs per platform.

  OniOS      iOS only
  OnAndroid  Android only
  OnMobile   iOS and Android behave the same, but differ from web
  OnWeb      Web browser only
  (none)     All platforms behave the same

Full examples:
  V1Login                          all roles, all devices
  V1CreateByAdmin                  admin only, all devices
  V1UpdateByUser                   user only, all devices
  V1ExportTaskByAdmin              background export, admin only
  V1UpdateByAdminOnMobile          admin only, iOS + Android
  V1UpdateByAdminManagerOnWeb      admin with Manager role, web only
  V1SyncTaskByUserOniOS            background sync, user only, iOS only

-------------------------- Translation Key Convention ------------------------------
------------------------------------------------------------------------------------

Format: NAMESPACE[key_name]

  NAMESPACE  - ALL_CAPS. Feature name (ADMIN, USER, BOOKING) or GLOBAL for shared strings.
  key_name   - all_lowercase_with_underscores. No camelCase, no dashes, no mixed case.

Examples:
  'GLOBAL[unauthorized]'              -> global shared string
  'ADMIN[invalid_login_credentials]'  -> Admin feature string
  'USER[profile_not_found]'           -> User feature string

Both are compiled into locales/en.json by yarn lang — never edit locales/*.json directly.

------------------------------------------------------------------------------------
--------------------- JavaScript File Structure (every .js file) -------------------
------------------------------------------------------------------------------------

Every JS file follows this top-to-bottom order:

  1.  Header comment — what the file is for + what its methods do
  2.  'use strict';
  3.  ENV variables (destructured from process.env)
  4.  Built-in node modules (fs, path, crypto, ...)
  5.  Third-party node modules (lodash, joi, moment-timezone, ...)
  6.  Services (from services/ — includes requiring the queue service, services/queue.js)
  7.  Helpers (from helpers/, constants included; and the feature's helper.js)
  8.  Models (const models = require('.../models'))
  9.  Queues (the queue INSTANCES via queue.get('XQueue') — right after models; the queue
      service itself is required up in Services, this is just the handles)
  10. Module-level variables/constants (optional)
  11. module.exports (before the methods — may sit below constants, not necessarily at the very top)
  12. The method definitions

Ordering within each import section: INCREASING LENGTH (clean visual ramp). Put plain whole-module
requires FIRST (whole module assigned to a variable), ordered by increasing length, THEN the
destructured imports, also ordered by increasing length.

  // third-party node modules
  const _ = require('lodash');
  const joi = require('joi');
  const moment = require('moment-timezone');

  // services — plain requires first (by length), then destructured (by length)
  const lang = require('../../../services/language');
  const queue = require('../../../services/queue');
  const { ERROR_CODES, errorResponse, joiErrorsMessage } = require('../../../services/error');

See the scaffolded action/task templates (app/feature.js output) for the canonical example.

------------------------------------------------------------------------------------
-------------------------- Module Export Convention --------------------------------
------------------------------------------------------------------------------------

Always place module.exports BEFORE the function definitions — NOT necessarily at the very top of
the file. Any constants or variables the file needs may sit above module.exports; what matters is
that module.exports comes before the methods (which works because function declarations are hoisted).
This gives every file a readable summary of what it exports right before the implementations.

  // ... env vars, requires, module-level constants may go here ...

  module.exports = {
    myFunction,
    anotherFunction
  };

  function myFunction() { ... } // END myFunction
  function anotherFunction() { ... } // END anotherFunction

For middleware or files that export a single function directly:

  module.exports = {
    requestId
  };

  function requestId(req, res, next) { ... } // END requestId

NOTE: Applies to all NEW code. Some early global files (e.g. services/error.js,
services/socket.js) predate this convention and place module.exports lower in the
file. They will be migrated over time — follow the convention above for anything new.

------------------------------------------------------------------------------------
-------------------------- Function End Comment Convention -------------------------
------------------------------------------------------------------------------------

Always close every named function with a comment: // END functionName

  function V1Login(req, res) {
    ...
  } // END V1Login

This makes it easy to find the end of long functions and scan a file quickly.
Apply to all functions: actions, tasks, helpers, services, middleware.

NOTE: Applies to all NEW code. Some early global files (e.g. helpers/cruqd.js,
helpers/schemas.js, services/error.js) predate this convention and are missing
// END comments. They will be migrated over time — follow the convention above
for anything new.

---------------------------- cruqd.js Improvement Notes ----------------------------
------------------------------------------------------------------------------------

The bracket-operator filter convention (age[gte]=18, createdAt[lte]=...) is a solid,
industry-standard pattern used by PostgREST, Strapi, Django REST Framework, and JSON:API.
The auto-conversion in middleware/args.js so actions never have to call it manually is
the strongest part of the implementation.

Three things worth adding when time allows:

1. UNIFY convertStringListToWhereStmt INTO parseUrlQueryFilter
   Currently there are two separate syntaxes for filtering: bracket notation for comparison
   operators and a separate function for IN lists. A new engineer has to learn both.
   Recommend adding an 'in' operator to parseUrlQueryFilter so status[in]=active,pending
   works the same way as age[gte]=18. One convention, one function.

2. ADD 'like' / 'ilike' OPERATOR
   String pattern matching (name[like]=john%) is extremely common in query APIs and will
   be needed eventually. Add Op.like and Op.ilike to the OPERATORS and SEQUELIZE_OPERATORS
   maps in cruqd.js.

3. CONSIDER RELATIONAL FILTERING
   Currently filters only work on the queried model's own columns. Filtering on associated
   models (e.g. user.firstName[like]=john) is a common need at scale. Non-trivial to add
   but worth considering as the API matures.

------------------------------------------------------------------------------------
---------------------------- Success Response Shape --------------------------------
------------------------------------------------------------------------------------

Every action returns a flat object. Always required: status (HTTP code) and success: true.
Everything else is endpoint-specific and named after what it contains.

  return {
    status: 200,
    success: true,
    admin: { ... },   // name after what you're returning
    token: '...'      // additional fields at the same flat level
  };

Do NOT nest payload under a data key:
  return { status: 200, success: true, data: { admin: { ... } } }  // ❌

Success status codes:
  200 - default for everything (reads, updates, queries, login, etc.)
  201 - action CREATES a new resource / database record (V1Create, signup)
  202 - action hands work off to a BACKGROUND JOB and returns an acknowledgement (e.g. jobId)

Why: every endpoint returns semantically different keys anyway — the client already
has to know what to expect. Nesting under data adds an extra destructuring level
everywhere without solving that inconsistency, and makes test assertions noisier.

------------------------------------------------------------------------------------
----------------------- HTTP Method & Request Args Convention ----------------------
------------------------------------------------------------------------------------

Use POST and GET only. Never use PUT, PATCH, DELETE, or other HTTP methods.
The action name carries the semantic meaning (V1Create, V1Update, V1Delete).
The HTTP method adds no value and forces engineers to remember an extra convention.

  POST  → default for all actions that send data or mutate state
  GET   → only when arguments fit cleanly in the URL query string

In actions, always use req.args — never req.body or req.query directly.
  req.args  → normalized by middleware/args.js (POST body or GET query, same interface)
  req.params → Express URL path segments (:id), used only when the route needs a
               dynamic path segment (e.g. third-party webhook callbacks)

--------------------- Flattened Ownership (Foreign Key) Convention -----------------
------------------------------------------------------------------------------------

Carry the OWNING entity's foreign key onto EVERY descendant table, not just the
immediate parent — even though it looks redundant.

  User -> UserOrder -> UserOrderItem

  UserOrder.userId           -> User.id
  UserOrderItem.userOrderId  -> UserOrder.id
  UserOrderItem.userId       -> User.id   <- ALSO here (looks duplicated, but required)

WHY:
  - "All items for this user" becomes a flat, indexed, join-free query:
      models.userOrderItem.findAll({ where: { userId } })
  - The owner id is also your SECURITY SCOPE — almost every query is "things belonging
    to this user". This is why multi-tenant SaaS puts accountId/orgId on every table.
  - Reads >> writes; joins are what hurt at scale.

DRIFT GUARDRAIL (REQUIRED) — use a COMPOSITE FOREIGN KEY so the redundant column
can never fall out of sync:
  1. Add UNIQUE (id, userId) on the parent (UserOrders).
  2. On the child (UserOrderItems), add a composite FK (userOrderId, userId)
     referencing UserOrders(id, userId).
  Postgres then rejects any item whose userId != its order's userId.

SCOPE RULE:
  - ALWAYS carry the top-level owner (userId) to every descendant.
  - Carry INTERMEDIATE ancestor keys onto a deep leaf ONLY when you actually query
    that leaf by that intermediate level. Don't add every ancestor key reflexively.

See README "Carry the Owner Foreign Key Down to Every Descendant" for full example.

------------------------------------------------------------------------------------
--------------------------------- Auth Conventions ---------------------------------
------------------------------------------------------------------------------------

Two-token model: short-lived ACCESS token (JWT, ~15m) + long-lived REFRESH token (opaque, ~60d).
One Passport strategy + one session table (UserSessions/AdminSessions) per user type.

1. ACCESS token: jsonwebtoken signed with ACCESS_TOKEN_SECRET; carries sub, type, tokenVersion;
   exp + iss + aud enforced in the JWT strategy. Returned in the response BODY (never a cookie).
   Sent on every request as: Authorization: jwt-<type> <token>
   AUDIENCE is per type AND client KIND: TOKEN_AUDIENCE = { USER: {WEB:'user-web', APP:'user-app'},
   ADMIN: {WEB:'admin-web', APP:'admin-app'} }. Client kind is the security boundary: 'web' = browser/cookie
   client, 'app' = native/token client (mobile AND native desktop). Signalled by X-Client header (web|app,
   default web), stored on the session, re-used on refresh. Strategy enforces audience as the ARRAY of a
   type's client audiences (Object.values), so any client of that type passes.
   PLATFORM (ios/android/ipados/macos/windows/web) is METADATA only — signalled by X-Platform header, stored
   on the session's `platform` column (and/or UserDevices). NEVER put OS/platform in the token audience.
2. REFRESH token: opaque 256-bit random string. NEVER store the raw token — store only its
   SHA-256 hash (hashToken). Set as httpOnly + Secure + SameSite=strict 'jwt-<type>-refresh' cookie
   (web) and returned in the body (mobile).
3. ROTATION + REUSE DETECTION: every /refresh revokes the presented session and issues a new one
   (linked via replacedBySessionId). Replaying an already-rotated token = theft → revoke ALL the
   user's sessions + bump tokenVersion.
4. INSTANT REVOCATION: bump <Type>.tokenVersion to invalidate all outstanding access tokens
   (logout-all, password change, compromise). Checked in the JWT strategy.
5. Token helpers live in helpers/logic.js (createAccessToken/createRefreshToken/hashToken);
   session mechanics in helpers/session.js (issue/find/rotate/revoke/revokeAll).
6. Adding a user type: add one entry to AUTH_TYPES in middleware/auth.js + a Passport strategy
   + a <Type>Session table. Everything else is registry-driven.
7. Auth actions are HTTP actions → return errorResponse(req, ...) on failure (401 for bad refresh token).

-------------------------------- Socket Conventions --------------------------------
------------------------------------------------------------------------------------

1. ALWAYS USE getIO() — never import `io` directly from socket.js.
   The module-level variable is null at require-time. getIO() returns the live instance.

     const { getIO, SOCKET_ROOMS, SOCKET_EVENTS, socketWrapper } = require('../../../services/socket');
     const io = getIO();

2. EMIT AFTER COMMIT — always emit socket events AFTER t.commit().
   If you emit before committing and the commit fails, clients receive an event for a
   record that doesn't exist in the database.

     await t.commit();
     io.to(room).emit(SOCKET_EVENTS.MESSAGE_CREATED, data); // ← after commit, always

3. SOCKET_ROOMS naming — ALL_CAPS_WITH_UNDERSCORES.
   Broadcast rooms: flat constant (GLOBAL, ADMIN, USER).
   Instance rooms: constant + socketWrapper(id) → 'USER<uuid>', 'CONVERSATION<uuid>'.

4. SOCKET_EVENTS naming — ALL_CAPS_WITH_UNDERSCORES, format: FEATURE_ACTION.
   Always import from socket.js constants — never hardcode the string.

5. CIRCULAR DEPENDENCY — actions imported by socket.js cannot require socket.js back.
   Use the context object pattern: socket.js passes { io, SOCKET_ROOMS, SOCKET_EVENTS,
   socketWrapper } as the second argument to those actions.
   HTTP actions that emit can use getIO() directly (no circular dep from that direction).

6. TESTING — never guard emits with if (NODE_ENV !== 'test').
   Use jest.spyOn(socket, 'getIO') to mock the io instance and assert on .to() and .emit().

---------------------------- helpers/schemas.js Convention -------------------------
------------------------------------------------------------------------------------

Reusable Joi schemas shared across more than one action live in helpers/schemas.js.

Rules:
1. NAMING: Every export must end in "Schema". Prefix describes what it validates.
     addressSchema, phoneSchema, dateSchema, numberComparisonSchema
2. PLACEMENT: Only add here if used in 2+ actions. One-off schemas stay inline in the action.
3. NO REQUIRED/OPTIONAL: Schemas are returned as functions so callers chain .required()
   or .optional() at the point of use. Never add those inside schemas.js.
4. COMPOSITION: Embed into action-level schemas at the call site via .keys() or direct embedding.
