# 設計哲學

Orbital Express 的每一個決策，最終都回歸到同一個問題：

**我們如何以現有的團隊，盡可能快速地交付最多功能——並在團隊不斷變動的過程中持續做到這一點？**

不是「什麼架構最令人印象深刻」，不是「什麼方案能撐到十億用戶」，也不是「大型科技公司都在用什麼」。只有一個問題：一支真實的團隊，能多快建出一個真實的產品，並長年持續下去。

這個問題有個出乎意料的答案。瓶頸幾乎從來都不是技術本身，而是人——新成員入門要多久、大家花多少時間爭論架構決策、讀懂陌生程式碼要多少時間、新人多快能做出真正有價值的貢獻。

Orbital Express 的目標，就是消除上述每一個瓶頸。

---

## 核心原則

這個 framework 裡每一個有主見的選擇，都直接體現了以下一條或多條原則：

### 1. 低學習曲線勝過強大功能

一個工程師第一天就能上手的 framework，遠比一個三年後才能解鎖某些理論能力的 framework 更有價值。我們優化的是「第一次 commit」到「第一個功能上線」之間的時間——對每位工程師，每一次都是如此。

### 2. 有主見才能消除爭論

當 framework 做出選擇，你的團隊就不必再做。每一個開放性問題——用什麼狀態管理、什麼 router、什麼資料夾結構、什麼 HTTP method、什麼命名規範——都是引發爭論的邀請函。Orbital Express 把這些問題全部關上。你把省下來的時間拿去開發功能。

### 3. 一致性勝過聰明

每個檔案看起來都和其他檔案一樣的程式碼，才是任何工程師第一天就能上手的程式碼。沒有人能理解的「天才個人解法」是一種負債。我們的目標是讓程式碼活得比任何一個人都久——包括你自己。

### 4. 將培訓成本降至趨近於零

每一個非標準的技術選型都是培訓成本。每多一層，就是新工程師在能做出貢獻之前必須理解的新事物。每一個需要深厚專業知識才能正確使用的工具，都會縮小你的招聘範圍、拉長入職時間。我們在任何可能的地方都把這些成本剔除掉。

### 5. 團隊速度，而非個人速度

衡量一支好的工程團隊，不是看最強的工程師能多快交付，而是看整個團隊能多快交付——包括上個月才加入的那位工程師。讓一個人快 10%、卻需要三週時間教會其他所有人的決策，是壞決策。

## 這些原則如何落地

這五條原則驅動了 framework 中的每一個選擇：

- [JavaScript，不是 TypeScript](/zh-TW/philosophy/javascript-not-typescript)——因為一個編譯層、一套型別系統、以及數個月的學習曲線，並不能提升團隊的交付速度。
- [Vue.js，不是 React](/zh-TW/philosophy/vue-not-react)——因為 React 的無主見特性意味著每位新成員都帶著自己的主見來，你花的時間是在裁判，而不是在開發。
- [單一 repo，不是微服務](/zh-TW/philosophy/one-repo)——因為五個 repo 意味著五套規範、五段入職體驗，以及凌晨兩點的跨服務除錯。
- [託管部署，不是 AWS](/zh-TW/philosophy/one-repo#deployment)——因為一支負責管理自建基礎設施的 DevOps 團隊，並沒有在開發功能。

而這一切背後：[這個 framework 為何存在的故事](/zh-TW/philosophy/the-origin)——十年昂貴的錯誤究竟教會了我們什麼。

---

::: tip 結論
最好的工程團隊，不是那個擁有最好技術的團隊。而是交付最多、學習最快、每次有新人加入都不會崩潰的團隊。為那樣的團隊而建。
:::
